## 슬라이드 3 — 설계 목표 달성 현황

제안서에서 설정한 3가지 핵심 목표의 달성 현황입니다.

첫째, **실시간 존재 여부 검증**은 PyPI와 npm API를 통한 메타데이터 조회로 구현 완료했습니다. Layer 1에서 패키지의 등록 여부, 등록일, 버전 수 등을 분석합니다.

둘째, **위험도 기반 점수 산출** 역시 완료했습니다. 제안서에서는 단일 계층이었지만, 현재는 Layer 1 메타데이터와 Layer 2 소스 분석을 결합한 2계층 판정 체계로 강화했습니다.

셋째, **HTML 리포트 생성**은 n8n 워크플로우를 통한 자동화 리포트와 마크다운 리포트를 모두 지원합니다.

## 슬라이드 4 — 추가 구현 기능

제안서 범위를 넘어 3가지 핵심 기능을 추가 구현했습니다.

**첫째, Layer 2 소스코드 분석입니다.** PyPI와 npm의 패키지 아카이브를 메모리에서 스트리밍하여, 다운로드나 설치 없이 악성 패턴 9종을 탐지합니다. base64와 exec의 조합, credential theft, postinstall 악용 등을 검사합니다.

**둘째, Chrome Extension입니다.** Claude, ChatGPT, Gemini 3개 AI 사이트에서 코드가 생성되는 순간 실시간으로 인라인 경고 패널을 표시합니다. 개발자가 코드를 복사하기 전에 위험을 알려주는 것이 핵심입니다.

**셋째, 자연어 패키지 감지입니다.** pip install 명령어가 없는 텍스트 응답에서도 백틱 코드, import 패턴, 130개 이상의 인기 패키지 사전 매칭을 통해 패키지를 감지합니다.

## 슬라이드 6 — 아키텍처

제안서에서는 단일 파이프라인이었지만, 현재는 3개 경로로 확장됐습니다.

**경로 1은 Chrome Extension입니다.** AI 사이트의 응답을 MutationObserver로 실시간 감지하고, background.js를 통해 FastAPI 서버에 분석을 요청한 뒤, 응답 바로 아래에 인라인 패널을 삽입합니다. SHA-256 기반 30분 캐시로 중복 API 호출을 방지합니다.

**경로 2는 CLI입니다.** slop_check.py로 소스 파일을 직접 분석하며, Rich 라이브러리로 터미널에 결과를 출력합니다.

**경로 3은 n8n 자동화입니다.** Gemini API를 통해 패키지를 추출하고 분석 후 HTML 리포트를 자동 생성합니다.

모든 경로의 분석 엔진은 Docker 컨테이너의 FastAPI 서버(포트 8001)에서 통합 처리됩니다.

## 슬라이드 7 — 기술 스택 상세

각 모듈의 기술 스택을 간단히 소개합니다.

Chrome Extension은 **Manifest V3** 기반이며, 사이트별로 분리된 content scripts로 Claude, ChatGPT, Gemini를 각각 지원합니다.

API 서버는 **FastAPI, Docker, uvicorn**으로 구성되어 안정적인 서비스를 제공하며, 소스코드 분석은 아카이브 메모리 스트리밍과 **9종의 악성 패턴 탐지**를 수행합니다.

자동화 파이프라인은 **n8n 워크플로우와 Gemini API**를 연동하고, 연구 모듈은 **OpenAI GPT-4o**를 활용한 비동기 실험과 SQLite 데이터베이스로 결과를 관리합니다.

## 슬라이드 9 — 분석 기준

위험도 판정은 3가지 핵심 기준으로 이루어집니다.

**편집 거리 활용**으로 기존 인기 패키지와의 유사성을 측정하며, 유사한 이름의 패키지가 선점됐을 가능성을 파악합니다.

**채널 침해 감지**에서는 합성 패턴, 편집 거리, 등록 시기 등 복합 요소를 종합하여 위험도를 다각적으로 판단합니다.

**등록 상태 확인**으로 PyPI와 npm 레지스트리에서 패키지의 등록 여부를 검증하여 환각 패키지를 식별합니다.

---

## 슬라이드 10 — 분석 강화 사항

제안서 대비 3가지 핵심 강화가 이루어졌습니다.

**레이어 구조 도입**으로 Layer 1에서 메타데이터를 분석하고, Layer 2에서 소스 분석을 통해 악성 패턴을 탐지하는 2단계 구조로 발전시켰습니다.

**신뢰도 할인 계수**를 도입했습니다. 오래된 안정적인 패키지가 정상적으로 eval이나 subprocess를 사용할 때 발생하는 오탐을 방지합니다. 등록 기간, 버전 수, 인기 패키지 여부에 따라 소스 점수에 곱셈 할인을 적용합니다. 예를 들어 pandas는 소스 점수 40점이지만, 등록 5953일 곱하기 0.2, 버전 114개 곱하기 0.7, 인기 패키지 곱하기 0.5로 최종 2점 LOW가 됩니다.

**추가 시그널**로 npm 생태계 혼동 패턴과 미등록 패키지의 양쪽 레지스트리 검증을 추가하여 분석의 정확성을 높였습니다.

---

## 슬라이드 11 — 판정 체계

최종 판정은 4단계로 분류됩니다.

**CRITICAL**은 80점 이상으로, 미등록 패키지이거나 악성 패턴이 다수 발견된 경우입니다. PyPI와 npm 양쪽 미등록 시 자동으로 80점 이상이 됩니다.

**HIGH**는 60~79점으로 의심 시그널이 복합적으로 존재하는 경우이며, **MEDIUM**은 30~59점으로 일부 시그널이 존재하여 주의가 필요한 상태입니다.

**LOW**는 29점 이하로, 정상적으로 등록된 안전한 패키지입니다.

핵심 변경 사항은 미등록 패키지의 기본 점수를 55점에서 70점으로 상향하여, 소스 분석이 불가능한 미등록 패키지에 더 높은 경각심을 부여한 것입니다.

## 슬라이드 13 — 파서 정확도 개선

LLM 응답에서 패키지명을 추출하는 과정에서 심각한 오검출 문제가 있었습니다.

예를 들어 코드 예제의 변수명인 `your_app_file_name`이나 `dummy_input`, 서브모듈 경로인 `torch.quantization`, 클래스명인 `MLFlowLogger`가 패키지명으로 잘못 추출되고 있었습니다.

이를 해결하기 위해 **3종의 필터**를 추가했습니다. Placeholder 필터, 서브모듈 경로 필터, PascalCase 클래스명 필터를 적용하여 오검출 패키지를 387개에서 358개로, 할루시네이션 판정을 43개에서 19개로 줄였습니다. 실제 할루시네이션율은 11.1%에서 5.3%로 절반 이하로 감소했습니다.

또한 Python AST와 Regex의 하이브리드 파싱 방식으로 정적 import과 동적 import을 모두 탐지하는 신뢰성 높은 파서를 구축했습니다.

## 슬라이드 15 — 로드맵 대비 현황

제안서에서 제시한 2가지 로드맵 항목의 현재 상태입니다.

**환각 패턴 데이터베이스 구축**은 진행 중입니다. GPT-4o 모델로 20개 질문에 대해 5회씩 총 100회 실험을 수행했으며, 결과를 SQLite에 저장하고 JSON과 마크다운 리포트를 자동 생성하는 파이프라인을 완성했습니다. 할루시네이션율 5.3%라는 구체적인 수치를 도출했습니다.

**개발 환경 실시간 연동**은 제안서에서 IDE 연동을 목표로 했으나, Chrome Extension으로 대체 구현했습니다. IDE가 아닌 AI 사이트 자체를 모니터링하는 방식으로, 오히려 더 이른 시점에서 위험을 탐지할 수 있게 되었습니다.

## 슬라이드 17 — Extension 안정화 계획

Chrome Extension 관련 향후 계획입니다.
